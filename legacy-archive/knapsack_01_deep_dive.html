0/1 Knapsack using Dynamic Programming — full line-by-line explanation with DP table

Core idea in one line
Each item is either taken fully (1) or not at all (0) — no fractions. Build a 2D table where knap[i][w] = max value using first i items with capacity w. Answer is at knap[n][W].
0/1 vs Fractional Knapsack
0/1 Knapsack (Lab 6)
Take item fully or skip. No fractions. Uses Dynamic Programming. O(n·W).
Fractional Knapsack (Lab 7)
Can take fraction of item. Uses Greedy (sort by value/weight ratio). O(n log n).

The DP recurrence — burn this in
If wt[i−1] > w:   knap[i][w] = knap[i−1][w]   (can't fit item i → skip it)
If wt[i−1] ≤ w:   knap[i][w] = max( knap[i−1][w],   val[i−1] + knap[i−1][w−wt[i−1]] )
                                ↑ skip item i        ↑ take item i (add its value, reduce capacity)
i−1 in val/wt because arrays are 0-indexed but table rows are 1-indexed. knap[i] = "using first i items". So item i is at index i−1 in the array.

How knap[3][50]=65 is reached
At i=3 (item3, wt=30, val=40), w=50:
wt[2]=30 ≤ 50 → can take it.
Option A — skip item3: knap[2][50] = 45
Option B — take item3: val[2] + knap[2][50−30] = 40 + knap[2][20] = 40 + 25 = 65
max(45, 65) = 65 ✓   Items taken: item2 (wt20,v25) + item3 (wt30,v40) = wt50, val65.
Viva questions
Why is the table (n+1)×(W+1)? — Row 0 = "no items" baseline (all zeros). Col 0 = "capacity 0" baseline (all zeros). The +1 handles these base cases cleanly without special-casing.
Why i−1 when accessing val[] and wt[]? — Table rows are 1-indexed (row i = first i items). Arrays are 0-indexed. So item i in the table = val[i−1] and wt[i−1] in the array.
Time and space complexity? — Both O(n·W). Table has (n+1)×(W+1) cells, each computed in O(1).
Why DP and not Greedy here? — Greedy (pick best ratio) doesn't work for 0/1 because you can't take fractions. A greedy pick might waste capacity. DP explores all combinations implicitly via the table.
What does knap[i][w] represent exactly? — Maximum value achievable using only the first i items with a knapsack of capacity exactly w.
Is this pseudo-polynomial? — Yes. O(n·W) depends on the value of W, not just the number of inputs. If W is huge, the table explodes.
